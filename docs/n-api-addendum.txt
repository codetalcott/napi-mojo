# advanced technical features and project-level concerns

These aren't immediate blockers but should be on our roadmap.

***
### ## Advanced N-API Features

Our current plan covers the fundamentals beautifully. The next layer of functionality required for many real-world addons includes:

* **Asynchronous Operations:** Node.js is single-threaded. Any Mojo function that takes more than a few milliseconds to run will block the event loop. We need a plan to use `napi_async_work` to allow Mojo to perform heavy computations on a background thread and return a `Promise` to JavaScript.
* **Object Wrapping:** A critical feature is associating a native Mojo `struct` with the lifecycle of a JavaScript object (e.g., `const myMojoObj = new MyMojoClass()`). This involves `napi_define_class`, `napi_wrap`, and `napi_unwrap`, and it's essential for building stateful addons.
* **Handling Buffers & TypedArrays:** For high-performance applications, you'll need to work with raw binary data. We should plan for helpers that can efficiently read from and write to Node.js `Buffer` and `TypedArray` instances without copying memory.
* **Mojo-to-JavaScript Error Propagation:** We have a plan for handling N-API errors *within* Mojo. We also need the reverse: a way to catch a native Mojo `Error` and properly convert it into a JavaScript `Error` object that can be thrown back to the user.

***
### ## Build & Automation (DevOps)

* **Cross-Platform Builds:** The current `build.sh` script is tailored for Unix-like systems (Linux, macOS). Building on **Windows** is a significant, separate challenge that typically requires Microsoft's C++ toolchain (MSVC). A robust build plan needs to address all three major platforms.
* **Continuous Integration (CI):** We should set up a CI pipeline (e.g., using GitHub Actions) early on. This pipeline should automatically build and run our Jest test suite on Windows, macOS, and Linux every time code is pushed to ensure we don't break compatibility.
* **Pre-built Binaries:** To make the library truly easy to use, users shouldn't have to compile it from source. We should plan to use a tool like `prebuildify` to publish pre-compiled binaries for various platform and Node.js version combinations.

***
### ## Project & Community Growth

* **Benchmarking:** One of our goals is performance. We should create a suite of benchmarks that compare `napi-mojo` performance against `napi-rs` (Rust), `node-addon-api` (C++), and pure JavaScript for a set of common tasks. This will validate our performance claims.
* **API Documentation Generation:** We need a strategy for generating user-facing API documentation from our Mojo docstrings, similar to `rustdoc`. This is key for user adoption.
